# FAC Packet Specification

## Overview

This document defines the serial packet format between the Python broker and the STM32 NUCLEO-G474RE firmware for the Secure Multi-Track Access system.

Goals:
- deterministic parsing on the MCU
- simple enough for coursework implementation
- robust against malformed serial input
- explicit support for local passkey flow and remote admin approval flow

## Recommendation

Use a framed binary packet with:
- start byte
- version byte
- message type byte
- flags byte
- request ID
- payload length
- payload
- CRC-16
- end byte

This is easier to debug than heavily compressed framing and simpler to implement first. COBS can be added later if needed.

## Packet Layout

| Field | Size | Description |
|---|---:|---|
| START | 1 byte | Fixed value 0x7E |
| VERSION | 1 byte | Protocol version, start with 0x01 |
| TYPE | 1 byte | Message type |
| FLAGS | 1 byte | Reserved for future use |
| REQUEST_ID | 2 bytes | Big-endian request identifier |
| LENGTH | 2 bytes | Payload length in bytes |
| PAYLOAD | N bytes | Message-specific payload |
| CRC16 | 2 bytes | CRC-16 over VERSION through PAYLOAD |
| END | 1 byte | Fixed value 0x7F |

## Message Types

| Hex | Name | Direction | Payload |
|---|---|---|---|
| 0x01 | PASS_TRY | Broker to MCU | ASCII digits for passkey |
| 0x02 | PASS_RESULT | MCU to Broker | 1 byte result code |
| 0x03 | REQUEST_ADMIN | Broker to MCU | optional reason code or empty |
| 0x04 | NONCE_ISSUED | MCU to Broker | 16-byte nonce |
| 0x05 | ADMIN_APPROVE | Broker to MCU | approval token |
| 0x06 | ADMIN_DENY | Broker to MCU | empty or reason code |
| 0x07 | STATUS_UPDATE | MCU to Broker | 1 byte state code |
| 0x08 | ERROR | MCU to Broker | 1 byte error code |
| 0x09 | LOCKOUT_NOTICE | MCU to Broker | remaining lockout seconds |
| 0x0A | PING | Broker to MCU | empty |
| 0x0B | PONG | MCU to Broker | empty |

## Result Codes

### PASS_RESULT payload

| Value | Meaning |
|---:|---|
| 0x00 | denied |
| 0x01 | granted |
| 0x02 | lockout active |

## State Codes

### STATUS_UPDATE payload

| Value | Meaning |
|---:|---|
| 0x10 | LOCKED_IDLE |
| 0x11 | VALIDATING_LOCAL |
| 0x12 | PENDING_ADMIN |
| 0x13 | UNLOCKED |
| 0x14 | DENIED |
| 0x15 | LOCKOUT |
| 0x16 | SYSTEM_FAULT |

## Error Codes

### ERROR payload

| Value | Meaning |
|---:|---|
| 0x20 | malformed packet |
| 0x21 | CRC mismatch |
| 0x22 | invalid state transition |
| 0x23 | nonce expired |
| 0x24 | nonce invalid |
| 0x25 | admin token invalid |
| 0x26 | unsupported version |

## Payload Definitions

### PASS_TRY
Payload:
- ASCII string of 4 to 6 numeric digits

Example payload:
- 31 32 33 34 for passkey 1234

### REQUEST_ADMIN
Payload:
- empty for basic implementation
- optional 1 byte source code in future versions

### NONCE_ISSUED
Payload:
- 16 random bytes generated by MCU

### ADMIN_APPROVE
Payload:
- 32 bytes HMAC-SHA256 output for nonce || action_code

Recommended action_code:
- 0x01 for APPROVE

Recommended token construction:
- HMAC_SHA256(admin_secret, nonce || 0x01)

### ADMIN_DENY
Payload:
- empty in the basic version
- future version may include 1-byte reason

### LOCKOUT_NOTICE
Payload:
- 1 byte remaining seconds if under 255 seconds

## Request ID Rules

- Broker generates request IDs for PASS_TRY and REQUEST_ADMIN
- MCU echoes the same request ID in the related response
- Nonce and admin approval for one admin session must use the same request ID
- Request IDs wrap after 65535

## Timeout Rules

- Admin nonce expires after 30 seconds
- Expired approval token must be rejected
- After timeout, the MCU must clear pending admin state and send STATUS_UPDATE back to LOCKED_IDLE

## Priority Rules

- If local passkey succeeds while admin request is pending, local success wins
- Pending admin nonce must be invalidated after local success
- Only one unlock action may occur for a given request lifecycle

## CRC Recommendation

- CRC-16/CCITT-FALSE
- polynomial 0x1021
- initial value 0xFFFF

## Example Flows

## Local passkey success

1. Broker sends PASS_TRY with request ID 100
2. MCU validates passkey
3. MCU sends PASS_RESULT granted with request ID 100
4. MCU sends STATUS_UPDATE unlocked
5. MCU later sends STATUS_UPDATE locked_idle

## Admin approval success

1. Broker sends REQUEST_ADMIN with request ID 200
2. MCU enters pending state and sends STATUS_UPDATE pending_admin
3. MCU sends NONCE_ISSUED with request ID 200
4. Admin GUI approves using token based on that nonce
5. Broker sends ADMIN_APPROVE with request ID 200
6. MCU verifies token and unlocks
7. MCU sends STATUS_UPDATE unlocked
8. MCU later sends STATUS_UPDATE locked_idle

## Parsing Rules on MCU

- ignore bytes until START is found
- reject packet if END is missing or length is invalid
- reject packet if CRC fails
- reject unknown TYPE values cleanly
- never trust payload length without bounds checks
- never use blocking waits for packet completion

## First implementation advice

Start with this exact packet format first.
Do not implement COBS, sockets, cloud sync, or multiple transport layers until the local serial protocol is working end to end.
